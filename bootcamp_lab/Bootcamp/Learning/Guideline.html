Recommendation + Action Plan

Clean Learning Plan for the Bootcamp

This page turns the guideline into a clearer path: focus on understanding the research workflow, not just finishing notebooks.

Recommendation

Your strongest direction is to treat this bootcamp as a training ground for research engineering, not a collection of exercises to complete. Since you already have software engineering experience, the key is to build judgment and independence.

1. Learn the pattern

Each notebook follows the same research workflow: problem → dataset → baseline → evaluation → training → failure → diagnosis → improvement.

2. Focus on modules

Every notebook can be understood through the same building blocks: data, tokenizer, model, loss, optimizer, training loop, evaluation, and inference.

3. Build intuition

Do not memorize APIs. Instead, test small changes and ask “what happens if I remove this line?”

4. Rebuild from memory

After each lab, try to recreate the core loop from memory before looking at the notebook again.

The goal is not to remember the code. The goal is to become someone who can read a paper and implement it independently.

Clean Plan

  1. Start with one notebook per day. Do not rush through it. Run it once, break it into modules, and understand why each part exists.
  2. Keep a one-page summary per lab. For each session, document topic, problem, key idea, pipeline, important APIs, experiments, failures, and open questions.
  3. Run controlled experiments. Change one variable at a time: learning rate, masking, tokenizer length, batch size, or data quality.
  4. Reimplement the important idea from memory. Write a tiny training loop, forward pass, loss, backward pass, and optimizer step without opening the notebook.
  5. Read one paper per lab. Use the abstract, figure 1, method, and conclusion to compare the notebook to the research paper.
  6. Build a mini research repo. Organize your work into folders such as data, tokenization, training, evaluation, alignment, RAG, RL, and projects.
  7. Turn labs into reusable components. Over time, create modules for tokenizer, training loop, evaluation, checkpointing, inference, and plotting.

Suggested Weekly Rhythm

  • Day 1: Understand the notebook structure and map the modules.
  • Day 2: Run one experiment and record the result.
  • Day 3: Recreate the core logic from memory.
  • Day 4: Compare the notebook to a paper or reference implementation.
  • Day 5: Refactor one part into a reusable module.

What to Keep in Mind

  • Spend more time changing and debugging notebooks than simply running them.
  • Build intuition through observation, not memorization.
  • Turn every lab into a reusable artifact for future work.
  • Measure progress by how independently you can reproduce the idea.
