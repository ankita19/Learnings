Section 1: Accuracy & Behavior

Ground truth comparison across 295 prod cases with human evaluator labels.

What does this section measure?

Exact Match
Every label must match perfectly. Strict — even selecting 2 of 3 correct labels counts as a miss.
Agreement
Did the model and human evaluator agree on the basic call: On Track vs. deficiency?
Catch Rate (vs Human)
Of deficiencies flagged by humans, what % did the model also catch? QEA's #1 metric.
Human Alignment
Of deficiencies flagged by the model, what % did humans agree with?
Confusion Matrix
The four outcomes: Defects Caught (both agree deficiency), Over-flagged (model only), Missed (human only), On Track Agreed (both agree On Track).
Behavioral Patterns
Strictness bias, verbosity, option selection patterns, inter-model agreement, and hardest cases — shown as subsections below accuracy.
Robustness
Agreement rates sliced by metadata dimensions (size bucket, source category, context window, PM patterns) — reveals blind spots across case types.

How are these metrics calculated?

Exact Match
GT labels == predicted labels (string identity after sorting and semicolon-joining).
Agreement
is_on_track(GT) == is_on_track(prediction). Includes ALL cases (On Track + deficiency).
Catch Rate (vs Human)
Caught / (Caught + Missed). Only cases where humans found a deficiency — On Track cases excluded.
Human Alignment
Caught / (Caught + Over-flagged). Only cases the model flagged. Low alignment may be over-flagging OR the model catching issues humans missed.
Confusion Matrix
Defects Caught = GT deficiency & model deficiency (TP). Over-flagged = GT On Track & model deficiency (FP). Missed = GT deficiency & model On Track (FN). On Track Agreed = both On Track (TN).
Behavioral Patterns
Strictness bias = model deficiency rate minus human base rate per category. Verbosity = word count of reason fields. Inter-model agreement = per case × category binary consensus across all models.
Robustness
Agreement is recomputed for each subset of cases (filtered by size, source, category, PM patterns, or input character count).

Behavioral Patterns

What does this section measure?

Deficiency Flagging Summary
Total deficiencies flagged by humans vs. each model. The gap shows how much stricter the model is.
Strictness Bias
How much stricter is the model compared to human evaluators per category? Positive = model flags more.
Verbosity
How long are the model's reasoning explanations? More words doesn't always mean better.
Option Selection Patterns
How many deficiency labels does the model pick per category on average?
Inter-model Agreement
How often do all 5 models agree with each other (independent of humans)?
Hardest Cases
Cases where every model disagreed with human evaluators — prime candidates for review.

How are these metrics calculated?

Deficiency Flagging Summary
Human Flagged = Caught + Missed (constant across models). Model Flagged = Caught + Over-flagged. From Section 1 confusion matrix.
Strictness Bias
Model deficiency rate minus human evaluator base rate, per category. E.g., +40% means the model flags deficiencies 40 percentage points more often than humans.
Verbosity
Word count of each 'reason' field in the model's response. Reported as mean, median, and max across all cases.
Option Selection Patterns
Average count of labels selected per category per case. Higher = model selects multiple deficiency types.
Inter-model Agreement
Per case × category: count how many of the 5 models agree on binary On Track vs deficiency. All agree / majority / no consensus.
Hardest Cases
Per case: across all models × categories, what % matched human evaluators. Bottom 10 cases shown.

Section 2: Quality Checks

Deterministic quality checks across all 310 cases. No LLM required.

What does this section measure?

Option Exclusivity
On Track must never appear with deficiency labels. Violation = model is confused. 0 = no violations.
Placeholder Echo
Model repeats {NAMEPII} redaction tokens from input. Cosmetic, not a security issue. 0 = no violations.
Absence Reasoning
Reason says 'no evidence of X' rather than citing specific case data. Lazy reasoning. 0 = no violations.
Data Sufficiency
Model says everything is fine despite having no emails or notes to evaluate. 0 = no violations.
Low Specificity
Reason lacks concrete details — no dates, actions, or case-specific references. 0 = no violations.
Summary Incoherence
Evaluation summary contradicts the labels chosen. Most common violation. 0 = no violations.
Structure Issues
Response is malformed or missing expected categories. Usually from truncation. 0 = no violations.
PII Compliance
Did the model leak real personal information (emails, phones, names)? Any real leak disqualifies the model. All models pass.

How are these checks detected?

Option Exclusivity
Split label string by ';'. Flag if On Track/No blockers co-exists with any deficiency label.
Placeholder Echo
Regex search for placeholder patterns ({NAMEPII}, {EMAILPII}, etc.) in the full response text.
Absence Reasoning
If response_value = On Track AND reason matches absence patterns ('no evidence', 'did not find', 'no indication').
Data Sufficiency
If emails and notes are both empty AND all categories are On Track AND evaluation_summary does not mention data limitations.
Low Specificity
If reason has < 15 words AND zero date references AND zero action verbs.
Summary Incoherence
If deficiency count >= 2 AND summary contains positive terms ('well', 'excellent', 'strong', 'commendable', etc.).
Structure Issues
Response must have exactly 5 category entries (matching v2 criteria). Also counts JSON parse failures.
PII Compliance
Regex patterns for real email addresses, phone numbers, and known synthetic names scanned across the full model response. Placeholder echo ({NAMEPII} etc.) tracked separately — not a real breach.

Section 3: Operational

Latency, cost, reliability, and caching metrics from run logs.

What does this section measure?

P95 Latency
How fast is the model? 95% of cases complete within this time.
Cost/Case
How much does it cost to evaluate one case?
Monthly @ 10K
Projected monthly cost at typical production volume (10,000 cases/month).
Token Distribution
Min/max/P95 for input and output tokens per case. Shows how token usage varies across cases of different sizes.
Prompt Composition
What makes up each API call: system prompt (7.4K chars, fixed), criteria (15.3K chars, fixed), and case data (varies). The fixed portions are cacheable.
Cache Efficiency
How much of the input is served from cache vs sent fresh. Higher cache hit = lower cost and faster response.
Response Size
How much output each model produces. Reasoning tokens (GPT-5 Reasoning) are internal chain-of-thought that aren't visible in the response but add to cost and latency.

How are these metrics calculated?

P95 Latency
95th percentile of response times from run log. Excludes failures and retries.
Cost/Case
Per-case cost = input token cost + output token cost. Claude: cache-aware pricing (uncached ×1.0, cache creation ×1.25, cache read ×0.10 of base input rate). GPT: flat input pricing (Azure cache discount not passed through CAPI). Rates from pricing.json.
Monthly @ 10K
Cost/Case × 10,000. Gives a practical budget comparison between models.
Token Distribution
Per-case token counts from run log. Input = system prompt + criteria + case data (all tokenized together). Output = model response tokens. Reasoning = internal CoT (GPT-5 Reasoning only, adds to cost but not visible in response).
Prompt Composition
Character counts from run log. System + criteria are identical across models (~22.7K chars, cacheable). Case data varies per case (avg ~27K, max 200K). Chars/Token: ~3.6 (Claude), ~4.0 (GPT).
Cache Efficiency
Cache read tokens = tokens served from prompt cache (cheaper, faster). Cache creation = first-time cache writes. Uncached = tokens processed fresh each time.

Section 4: Decision Matrix

What does this section measure?

Weighted Score
A single number ranking each model across 6 dimensions using raw performance percentages. Higher = better overall fit for QEA.
Catch Rate (30%)
Catching quality issues is QEA's core mission — highest weight.
Agreement (25%)
How often the model's binary call matches human evaluators.
Latency P95 (15%)
Faster response = higher score. Near-real-time evaluation matters.
Cost Efficiency (15%)
Lower cost per case = higher score.
Quality Checks (10%)
Deterministic quality pass rate — models with fewer violations (incoherence, structure issues) score higher.
Reliability (5%)
Success rate — model must consistently return valid responses. Low weight: all models >97%.

How is the score calculated?

Higher-is-better dimensions
Catch rate, agreement, quality, reliability use the raw percentage directly (e.g., 90.9% catch rate → score 90.9).
Lower-is-better dimensions
Cost and latency use ratio scoring: (best model / this model) × 100. Cheapest model = 100; 2× more expensive = 50.
Weighting
Dimension scores are multiplied by weights (30% + 25% + 15% + 15% + 10% + 5% = 100%).
Final Score
Sum of all weighted dimension scores. Scores are stable — adding or removing a model does not change other models' scores (except cost/latency ratios if the cheapest/fastest model changes).
